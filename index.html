02 · Planteamiento

El problema

Muchos sistemas tratan los datos como si fueran independientes y pierden la información más valiosa: el orden. Predecir la demanda eléctrica o entender un patrón sísmico exige modelar la dependencia temporal.

  • Sobrecarga de la red eléctrica si la demanda no se anticipa.
  • Ausencia de patrones detectables en eventos sísmicos esporádicos.
  • Modelos clásicos que ignoran la secuencialidad.
time_series.py 
import torch.nn as nn

class LSTMModel(nn.Module):
    def __init__(self, input_size, hidden_size):
        super().__init__()
        self.lstm = nn.LSTM(
            input_size=input_size,
            hidden_size=hidden_size,
            batch_first=True
        )
        self.fc = nn.Linear(hidden_size, 1)

    def forward(self, x):
        out, _ = self.lstm(x)
        return self.fc(out[:, -1, :])

06 · Análisis de riesgos

10 riesgos, 4 categorías

Cada riesgo está anclado en evidencia concreta del código y de los resultados. Filtra por categoría para explorarlos.

CRÍTICA

Distribución desbalanceada

Los días sin sismos quedan codificados como cero, sesgando la distribución.

Resolución: Propagación estadística del día anterior e indicador binario de actividad.
Datos · Sísmico
CRÍTICA

Regresión hacia la media

El modelo aprende a predecir el promedio (~0.13) ignorando eventos reales.

Resolución: Optimización de datos, aumento a 100+ épocas con early stopping y arquitectura de 14 días.
Modelo · Sísmico
ALTA

Data leakage

MinMaxScaler aplicado al dataset completo antes del split temporal.

Resolución: MinMaxScaler ajustado exclusivamente en el conjunto de entrenamiento (X_train).
Datos · Energía
ALTA

Sin conjunto de validación

Imposible aplicar early stopping ni ajustar hiperparámetros con rigor.

Resolución: Implementado split temporal 70% entrenamiento / 15% validación / 15% prueba.
Diseño · Ambos
ALTA

Ventana insuficiente

7 días no alcanza para capturar precursores sísmicos.

Resolución: Ventana ampliada exitosamente de 7 a 14 días para capturar precursores sísmicos.
Diseño · Sísmico
MEDIA

Sin regularización

No hay Dropout, weight decay ni Batch Normalization en el modelo.

Resolución: Integrado weight decay (1e-5) y ReduceLROnPlateau en optimizador Adam.
Modelo · Sísmico
MEDIA

Sobreajuste potencial

Sin curva de validación no se confirma la generalización del modelo.

Resolución: Monitoreo activo de pérdidas de validación y aplicación de early stopping.
Modelo · Energía
MEDIA

Sin reproducibilidad

No se fijan semillas aleatorias para PyTorch ni NumPy.

Resolución: Establecidas semillas fijas en PyTorch y NumPy (manual_seed(42)).
Operacional · Ambos
MEDIA

Modelo no guardado

No se persiste con torch.save al finalizar el entrenamiento.

Resolución: Persistencia automática de pesos (.pth) y metadatos (.json) al terminar.
Operacional · Ambos
BAJA

Carga computacional CPU

64,850 pasos forward+backward sobre CPU saturan la RAM.

Resolución: Submuestreo de datos (1 de cada 4) y uso de memoria mapeada (mmap_mode).
Operacional · Energía
07 · Arquitecturas

Variantes de LSTM exploradas

Cada arquitectura aporta un compromiso distinto entre capacidad, costo y precisión.

A

LSTM Simple

Una sola capa que procesa la secuencia. Modelo base, eficiente y fácil de entrenar.

  • ✓ Bajo costo
  • ✗ Capacidad limitada
B

Stacked LSTM

Múltiples capas apiladas que aprenden representaciones cada vez más profundas.

  • ✓ Patrones complejos
  • ✗ Riesgo de sobreajuste
C

Bidirectional LSTM

Procesa la secuencia en ambas direcciones para enriquecer el contexto.

  • ✓ Mayor contexto
  • ✗ No apto para forecast
D

CNN-LSTM

Combina convoluciones para extraer patrones locales con LSTM temporal.

  • ✓ Patrones híbridos
  • ✗ Más hiperparámetros
